## Bug 1 — `blog-crawler` creates a duplicate run instead of using the orchestrator's

### Reproduction

1. Orchestrator calls `register_run` and gets `run_id = 2026-04-25T10-44-57`.
2. Orchestrator dispatches `Task(subagent_type="blog-crawler", ...)` with `run_id` in the prompt and absolute paths from `register_run`.
3. After the crawler returns, `list_artifacts(run_id="2026-04-25T10-44-57", namespace="articles")` returns **`[]`**.
4. A *new* run `2026-04-25T10-45-39` exists on disk with all 3 articles written to it.
5. The new run's `state.json` has `peec_project: {id: null, mode: "generic"}` — the Peec association passed via the orchestrator was lost.

### Root-cause hypothesis

The blog-crawler agent prompt or its bundled instructions tell it to call `register_run` itself, or it calls `register_run` as a fallback when it cannot interpret the orchestrator's `run_id`. The crawler then writes everything to the run it created, ignoring the orchestrator's `run_id`.

Inspect: `agents/blog-crawler.md` for any `register_run` mention or any "create a run" instruction. The fix is to make the crawler **strictly receive `run_id` from its prompt and never call `register_run`**.

### Fix scope

- `agents/blog-crawler.md`:
  - Remove any instruction or example that calls `register_run`.
  - Add explicit non-negotiables: "You MUST receive `run_id` from the orchestrator prompt. You MUST NOT call `register_run`. If `run_id` is missing or unparseable, abort with a clear error in the response."
  - Validate that the crawler reads `run_id` from the prompt and uses it for every dashboard MCP write.
- `dashboard/server.py`: if `register_run` can be called by sub-agents at all, add a guard: a sub-agent context (or a flag passed by the orchestrator) is forbidden from creating a new run. The simplest implementation is an env var or session header that the orchestrator sets and the dashboard MCP rejects from sub-agent contexts. If that guard is not feasible, this bug must be fixed at the agent-prompt layer alone — but ALSO make the dashboard log a warning when `register_run` is called within 60 seconds of a previous successful `register_run` for the same `blog_url`.

### Acceptance test

Add to `tests/`:

```text
tests/bug_01_crawler_run_id_discipline_test.md
```

Manual reproduction sequence:

1. Call `register_run(blog_url, peec_project_id="or_e15a6ac5-...")` and capture `run_id_A`.
2. Dispatch `blog-crawler` with `run_id=run_id_A`, `blog_url`, `max_articles=3`, and the absolute paths from the register_run response.
3. After crawler completes:
   - `list_artifacts(run_id=run_id_A, namespace="articles", suffix=".json")` MUST return ≥1 artifacts.
   - There MUST be no other run created in `runs/` after the timestamp of the `register_run` call.
   - State at `runs/{run_id_A}/state.json` MUST contain `peec_project.id == "or_e15a6ac5-..."`.

Pass criterion: all three asserts pass on a clean run against any blog with ≥3 articles.

## Bug 2 — `evidence-builder` agent type is not registered in the Claude Code session

### Reproduction

1. Plugin v0.5.3 (and the in-flight v0.5.6) ship `agents/evidence-builder.md` with valid frontmatter:
   ```
   name: evidence-builder
   description: ...
   model: sonnet
   maxTurns: 18
   ```
2. The Claude Code resolver in a fresh session lists 6 of the 7 plugin agents. Missing: `evidence-builder`.
3. Available list seen on 2026-04-25: blog-crawler, competitor-crawler, generator, peec-gap-reader, recommender, voice-extractor.
4. Calling `Task(subagent_type="ai-search-blog-optimiser:evidence-builder", ...)` errors: `Agent type 'ai-search-blog-optimiser:evidence-builder' not found.`

### Root-cause hypothesis

Three plausible causes — each must be ruled out:

1. **Manifest mismatch**: `.claude-plugin/plugin.json` may declare an explicit `agents` list that doesn't include `evidence-builder`. Verify `plugin.json` either omits `agents` (so all agent files are auto-discovered) or includes `evidence-builder` explicitly.
2. **Agent file frontmatter schema mismatch**: compare frontmatter shape against agents that DO load (e.g. `recommender.md`). Look for missing required fields, malformed YAML, or non-standard keys.
3. **Plugin install/cache freshness**: the live install at `~/.claude/plugins/marketplaces/local-desktop-app-uploads/ai-search-blog-optimiser/` may have an older copy that pre-dates the agent. Compare its `agents/` listing.

### Fix scope

- Audit `.claude-plugin/plugin.json` for an `agents` list. If present, ensure `evidence-builder` is listed. Best practice: omit the explicit list so all `agents/*.md` files auto-load.
- Diff `agents/evidence-builder.md` frontmatter against `agents/recommender.md` frontmatter. Bring fields and ordering into parity. Specifically verify:
  - `name`, `description`, `model`, `maxTurns` are all present and valid.
  - No surplus keys that the resolver might reject.
  - YAML is valid (no tab vs space drift, no trailing colons without values).
- If the live cache is stale, document the rebuild step in `README.md` install section: how to rebuild the cache, how to verify a new agent is registered.

### Acceptance test

Add to `tests/`:

```text
tests/bug_02_evidence_builder_registered_test.md
```

Manual reproduction sequence:

1. From a Claude Code session with the plugin enabled, list available agent types via the Task tool. Confirm `ai-search-blog-optimiser:evidence-builder` is in the list.
2. Call `Task(subagent_type="ai-search-blog-optimiser:evidence-builder", description="smoke", prompt="...minimal valid input block...")` and confirm the agent runs without an `Agent type not found` error.
3. Assert the smoke prompt produces a valid evidence JSON output.

Pass criterion: all three asserts pass.

## Bug 3 — Dashboard MCP `write_json_artifact` and `read_json_artifact` namespace enum is missing `evidence`

### Reproduction

1. Even if `evidence-builder` ran, the playbook tells it to write to `evidence/{slug}.json`.
2. The dashboard MCP `write_json_artifact` tool accepts `namespace` from this enum:
   ```
   ["site", "articles", "recommendations", "optimised", "gaps", "competitors", "peec_cache"]
   ```
   `evidence` is not in the list.
3. The `read_json_artifact` enum is:
   ```
   ["run", "site", "articles", "recommendations", "optimised", "raw", "gaps", "competitors", "peec_cache", "media"]
   ```
   `evidence` is also missing here.
4. The `list_artifacts` enum has the same gap.

### Root-cause hypothesis

The v0.5.x cycle introduced the evidence stage in the playbook but the corresponding namespace was never added to the dashboard MCP tool schemas.

### Fix scope

- `dashboard/server.py` (or wherever the dashboard MCP tool definitions live):
  - Add `"evidence"` to the namespace enums for `write_json_artifact`, `read_json_artifact`, and `list_artifacts`.
  - Wire the namespace to the canonical filesystem path: `{run_dir}/outputs/evidence/`.
  - On `register_run`, ensure `outputs/evidence/` is scaffolded alongside the other namespace directories so writes don't fail on a missing parent.

### Acceptance test

Add to `tests/`:

```text
tests/bug_03_evidence_namespace_test.md
```

Manual reproduction sequence:

1. Call `register_run(...)` and capture the response. Verify `outputs/evidence/` exists on disk.
2. Call `write_json_artifact(run_id, namespace="evidence", relative_path="smoke.json", data={"k": "v"})`. MUST succeed.
3. Call `read_json_artifact(run_id, namespace="evidence", relative_path="smoke.json")`. MUST return the data.
4. Call `list_artifacts(run_id, namespace="evidence")`. MUST return the smoke artifact.

Pass criterion: all four asserts pass.

## Bug 4 — Generator references a `validate_article` tool that does not exist in the dashboard MCP

### Reproduction

1. Pipeline skill `skills/blog-optimiser-pipeline/SKILL.md` (Stage 7) tells the generator: *"Call `validate_article` after draft files are written. The validator overwrites the manifest and refreshes draft stage truth."*
2. Generator agent prompt repeats: *"Call `validate_article` after draft files are written."*
3. On the live 2026-04-25 run, one of the three generators reported: *"No `validate_article` tool exists in the dashboard plugin. I'll proceed by running the re-audit manually and then pushing state directly."*
4. The "manifest quality gate" the playbook references therefore was never actually evaluated; the audit-after scores were self-reported by each generator.

### Root-cause hypothesis

Either:

- The `validate_article` tool was planned but not implemented in `dashboard/server.py`, OR
- It was implemented under a different MCP tool name and the playbook + generator prompt were never updated.

### Fix scope

Pick ONE of these two paths and execute fully:

**Path A — implement the missing tool**:

- Add `validate_article` to `dashboard/server.py` as a real MCP tool.
- Inputs: `run_id`, `article_slug`. Outputs: a manifest JSON containing the rubric scorecard, pass/fail per dimension, and an aggregate pass/fail at the documented threshold (default 32/40 per the v0.5.3 pipeline).
- The validator MUST read the optimised draft files (`optimised/{slug}.md`, `optimised/{slug}.html`, `optimised/{slug}.schema.json`), apply the rubric defined in `references/geo-article-contract.md`, and write the manifest to `optimised/{slug}.manifest.json`.
- Update `state.json` `articles[i].stages.draft` with `quality_gate` and `audit_after` fields based on the manifest result.

**Path B — drop the tool reference**:

- Remove all mention of `validate_article` from `skills/blog-optimiser-pipeline/SKILL.md` and `agents/generator.md`.
- Replace with explicit instructions: "After writing draft files, the generator MUST compute a self-rubric scorecard and write it as `optimised/{slug}.manifest.json` with fields `audit_before`, `audit_after`, `breakdown`, `quality_gate` (pass|fail at >=32/40)."
- Update `state.json` write contract so the generator persists the audit_after directly, and document that the orchestrator does NOT have a server-side gate.

Recommended path is **A** (fewer trust assumptions, matches the v0.5.3 changelog claim about manifest-driven draft visibility), but B is acceptable if Path A scope is large.

### Acceptance test

Add to `tests/`:

```text
tests/bug_04_validate_article_test.md
```

For Path A:

1. Run the full pipeline against any 1 article. After generator completes:
2. Confirm `optimised/{slug}.manifest.json` exists and parses as JSON with the rubric shape.
3. `state.json articles[0].stages.draft.quality_gate` MUST equal `"passed"` or `"failed"` (not absent, not `"unknown"`).

For Path B:

1. Search the codebase for `validate_article`. Zero matches.
2. Generator prompt now contains an explicit self-rubric write contract.
3. Manifest file is still written and `state.json` is still updated.

Pass criterion (either path): all asserts pass.
